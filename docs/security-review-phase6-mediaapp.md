# Phase 6 (Media App) — Security Review & Sign-off (Packaging)

**Scope:** the Phase-6 change set on branch `media-app` — commit range `771a4d8..HEAD` (`771a4d8` = the Phase-5 baseline). Phase 6 adds the first runnable **`portable-server`** launcher (the server was previously a library only), formalizes the **client portable folder layout**, and adds **packaging scripts**. It makes the stack "just run" for testing while preserving prod parity.

**Method:** TDD, subagent-driven on Opus, per-task two-stage review (a dedicated security/composition review of the launcher), all findings fixed before acceptance, plus a final holistic review. Verification artifacts: the launcher unit tests (config/layout/pki/bootstrap); `crates/portable-server/tests/boot_smoke.rs` (the dev server boots over real loopback TLS and a client runs `POST /v1/bootstrap` — wrong-secret 401, right-secret 201); and the green gate suite (clippy `-D warnings`, `cargo deny`, `cargo audit`, `MAXSECU_PG_OPTIONAL=1 cargo test --workspace`, `cargo build --release -p maxsecu-portable-server`).

**Verdict:** **PASS** — no Critical, High, or Medium findings. `portable-server` only **composes and supervises** the unchanged secret-free server (no new crypto, no new endpoint); the dev directory-signing (D5) key and dev bootstrap secret are runtime-generated, security-degraded, and loudly labeled dev-only; no secret is baked into the binary; the prod profile fails honestly. The environment-blocked deferrals (real PostgreSQL bundling, Authenticode signing, Tauri GUI bundle) are documented with their wiring hooks — they are packaging/ops gaps, **not** security gaps.

---

## 1. What Phase 6 added (and did not)

- **No core change.** `portable-server` consumes `maxsecu_server::{serve, router, AppState, AuthConfig, AuthService, MemoryStore, FsBlobStore, NullAuditSink}` as-is. No new server route, no new cryptography. The only `client-app` change is the additive `layout` module.
- **`crates/portable-server`** (NEW bin+lib): `config` (dev/prod profile from env), `layout` (portable folder), `pki` (dev self-signed pinned cert + rustls `ServerConfig`), `bootstrap` (one-time bootstrap secret + dev D5 from a persisted seed), `run` (`prepare` composes `AppState` + binds + serves; `run` prints the secret once + exports client pins), `main` (thin shell), and the boot smoke test.
- **`client-app/src/layout.rs`**: `ensure_portable_layout` creates `config/keystore/index/cache/logs` on startup (best-effort).
- **`packaging/`**: `package.sh`/`package.ps1` (build release artifacts + lay out the portable folders, with guarded deferred-ops steps) + `README.md`.

## 2. Per-area findings & dispositions

| # | Area | Finding | Severity | Disposition |
|---|---|---|---|---|
| 1 | **Secret-free server preserved** | **Sound (✓).** `AppState` carries only the pinned D5 **public** key (`with_directory_pub`) and the bootstrap-secret **hash** (`with_bootstrap_secret_hash`) — the Phase-2 coarse-authz inputs. The `DirectorySigner` holding the dev D5 **private** seed is local to `bootstrap::ensure_dev_d5`, used only to derive `public_key()`, then dropped; it never enters `AppState`. The server it serves verifies under the pinned public key only and stores opaque client-verified bytes — unchanged. | Info | Accepted (correct). |
| 2 | **Dev D5 + bootstrap secret are dev-only, runtime-generated** | **Sound (✓).** Both are generated by the CSPRNG into the data dir on first run — **nothing is baked into the binary** (`stack.md §5.1`). The dev D5 is persisted as a 32-byte seed so the pinned public key is stable across restarts (a client that pinned it stays valid), and is loudly labeled "DEV ONLY — replace with the offline ceremony key in production" in the module docs and the console banner. In production the D5 private half is offline (the air-gapped ceremony); the launcher only ever needs the public key pinned. | Info | Accepted (correct). The dev D5 seed (`config/d5_secret.bin`) is a cleartext private Ed25519 seed — the README documents that the dev data dir must not be shared. Dev-posture, not a prod path. |
| 3 | **Bootstrap secret handling** | **Sound (✓).** Generated once, **printed once** to the operator console, and stored **only as `sha256`** (a 32-byte marker) — never in cleartext on disk, matching the Phase-2 design. A subsequent run prints "already bootstrapped" and reuses the stored hash. The launcher wires the hash into `AuthConfig.with_bootstrap_secret_hash`, so the server gates the bootstrap window exactly as Phase 2 reviewed (window open only until the first published binding; secret compared hash-to-hash, no oracle). | Info | Accepted (correct). |
| 4 | **Prod profile is honest** | **Sound (✓).** When `DATABASE_URL` is set (Prod), `prepare` returns a clear error naming the required Postgres + injected cert/sink + `schema.sql` self-apply — it does **not** fabricate a degraded "prod" server with dev artifacts. The dev profile (no `DATABASE_URL`) runs on `MemoryStore` + `FsBlobStore` + a self-signed cert. The prod wiring (PgStore + schema) is the documented, exercised-with-a-DB path. | Info | Accepted (correct). |
| 5 | **TLS** | **Sound (✓).** The self-signed cert uses the **aws-lc-rs** provider only (no ring/openssl — `cargo deny` ban intact), mirroring the audited `test_pki()` construction; TLS 1.3 defaults. The launcher binds **loopback-only** (`127.0.0.1`) — the security-conservative default for a dev launcher (not network-exposed). The cert is exported for the client to pin (the auto-connect scenario), and the smoke test exercises the real handshake + chain-of-trust + `localhost` SAN verification. | Info | Accepted (correct). |
| 6 | **Packaging scripts** | **Sound (✓).** `package.{sh,ps1}` build the cargo artifacts + lay out the portable folders. The three environment-blocked steps (Tauri GUI bundle, Authenticode signing, PostgreSQL bundling) are **guarded** — they run only if the tool/cert/dist is present (`cargo tauri`, `signtool`+`MAXSECU_SIGN_CERT`, `MAXSECU_PG_DIST`) and otherwise print a clear "DEFERRED (not available): …" notice without failing the build or **fabricating a signed/bundled artifact**. No false claim of a signed exe. | Info | Accepted (correct). |
| 7 | **Client portable layout** | **Sound (✓).** `ensure_portable_layout` creates the canonical `config/keystore/index/cache/logs` on startup, best-effort (a failure logs and does not crash; the individual writers also create their parents). Holds no secret beyond the existing at-rest ciphertext blob/keystore. | Info | Accepted (correct). |

## 3. Threat-model coverage (Phase-6-relevant)

- **Baked-in secret in a distributed binary:** closed — all secrets (cert key, D5 seed, bootstrap secret) are generated at runtime into the data dir (dev) or injected at runtime (prod); none is in the binary.
- **Operator mistakes a dev launcher for production:** mitigated — the dev D5 + bootstrap secret are loudly labeled dev-only (docs + console + README); the prod profile is a distinct, honest path.
- **Fabricated/false code-signing or PG bundling:** closed — the packaging scripts never produce a signed exe or a PG bundle when the prerequisite is absent; they print a DEFERRED notice.
- **Bootstrap-secret disclosure at rest:** closed — only `sha256` is stored; cleartext is printed once and never persisted.
- **Network exposure of the dev server:** mitigated — loopback-only bind by default.

## 4. Residuals / deferrals (intentional, environment-blocked — NOT security gaps)

- **Real PostgreSQL binary bundling + supervision** — needs PG binaries + a bundler; the prod profile wires `PgStore`/`schema.sql` behind `DATABASE_URL`, the dev profile runs on `MemoryStore`, and `package.*` has the guarded `MAXSECU_PG_DIST` step. A CI/ops packaging step.
- **Authenticode signing** (client + server exe) — needs a code-signing cert; `package.*` has the guarded `signtool`/`MAXSECU_SIGN_CERT` step. (Existing MaxSecu deferral.)
- **Tauri GUI bundle** (`cargo tauri build`) — the Tauri CLI is not installed here; `cargo build` produces the `client-app` binary; the bundled WebView2 installer is the guarded `cargo tauri` step.
- **Reproducible-build flags + transparency-logged release** — documented in `packaging/README.md`; the offline release-signing key + release transparency are existing Phase-7 ops deferrals.
- **In-process dev sink** (`sink-server`) wired into the dev launcher — optional; dev uses `NullAuditSink` (the injection point is noted). Cosmetic launcher items: a redundant `Layout::ensure` in `run`; loopback-only bind (no bind-address knob yet); dev D5 seed file permissions. None affect security.
- **P7 video** remains the separate, gated, security-reviewed effort (spec §10.7 / D-B).

## 5. Conclusion

**PASS.** Phase 6 delivers a runnable portable server that "just runs" for testing (boots from an empty data dir over real TLS, prints a one-time bootstrap secret, and a client bootstraps against it — proven by the smoke test) while preserving the zero-knowledge core: it composes the unchanged secret-free server, keeps the D5 private key out of the server, bakes no secret into the binary, stores the bootstrap secret only hashed, and fails the prod profile honestly. The packaging scripts build what the environment allows and guard the deferred-ops without fabricating signed/bundled artifacts. No Critical/High/Medium issues; the deferrals are documented packaging/ops gaps with their wiring hooks in place.
