# Phase 3 (Media App) — Security Review & Sign-off (Browse + View)

**Scope:** the Phase-3 change set on branch `media-app` — commit range `311ecfd..00100ad` (`311ecfd` = the Phase-2 baseline). Phase 3 adds the browse/view experience: feed/library listing, per-item card decryption (title + thumbnail), client-side title+tag search over a local **encrypted** index, the image + blog **viewer**, and the fetch/decrypt feedback layer — implemented as `client-core` (one pure addition) + `client-app` orchestration + a vanilla-TS Web-Component UI, with an end-to-end exit-gate test over real TLS.

**Method:** every task was built TDD, subagent-driven on Opus, with a per-task two-stage review (spec-compliance then code-quality), all findings fixed before acceptance, plus a final holistic review. Verification artifacts: the per-file unit tests; `crates/client-app/tests/browse_view_e2e.rs` (the 5-gate exit test over loopback TLS 1.3, MemoryStore + FsBlobStore); and the green gate suite (clippy `-D warnings`, `cargo deny`, `cargo audit`, `MAXSECU_PG_OPTIONAL=1 cargo test --workspace`, UI typecheck + build).

**Verdict:** **PASS** — no Critical, High, or Medium findings remain. Phase 3 introduces **no new server cryptography** and preserves the secret-free / zero-knowledge / client-re-verified model. One Critical content-substitution bug was found by the code-quality review during Task 9 and fixed before acceptance (see §3); the fix is proven by an e2e gate. Documented residuals (§4) are security-neutral.

---

## 1. What Phase 3 added (and did not)

- **No server change.** Phase 3 reuses the existing, already-reviewed server endpoints unchanged: `GET /v1/files` (D35 listing), `GET /v1/files/{id}` (§8.5 file view), the chunk `GET` endpoints, and `GET /v1/directory/by-id/{id}` / `/{username}`. No new route, no new server-held secret, no new server crypto.
- **One pure `client-core` (TCB) addition:** `verify_and_open_headers` — a header-only open that runs the **same** §12.5 ladder via the existing private `verify_header`, then decrypts only the non-content (small) streams. It is strictly a *subset* of the audited streaming path (`verify_and_stream_content`) — it fetches/releases no content and adds no new verification logic or bypass. Covered by three TCB tests (header opens without content; forged manifest → `ManifestSignature`; tampered small-stream → digest/framing rejection).
- **`client-app` orchestration** (outside the cryptographic core): pin the offline D5 key from config; resolve+verify a file's author binding under that pinned key; parse the opaque §8.5 view into a `DownloadBundle`/`StreamHeader`; run the `client-core` verify ladder; shape the result into render-ready DTOs. A local **encrypted** search index. A typed `FetchPhase` feedback machine.
- **UI** (vanilla TS Web Components, outside the TCB): feed grid, self-decrypting media cards, the image/blog viewer, shared feedback components, a serial queue.

## 2. Per-area findings & dispositions

| # | Area | Finding | Severity | Disposition |
|---|---|---|---|---|
| 1 | **TCB boundary (what crosses the Tauri seam)** | **Sound (✓).** The browse/view commands return only render-ready DTOs: `FeedEntryDto` (listing structure/sizes only — no values), `CardDto` (title/tags/thumbnail/`author_fp`/`mine`/`recovery_ok`), `OpenedContentDto` (the **content being displayed** — a canonical PNG as base64, or sanitized blog text — plus title/tags/ticks), and `SearchHit`s. A private key, session token, `Dek`, `WrappedDek`, grant/manifest/genesis interior, `ancestor_grants`, or the raw search index **never** crosses. The decrypted content reaching the WebView to be *displayed* is the product, not a leak; the prohibition is on key/crypto material and signed-record interiors, which stay in the `client-app`/`client-core` TCB. | Info | Accepted (correct). |
| 2 | **Directory trust — D5 re-verification** | **Sound (✓).** Every served author/owner binding is re-verified against the **pinned offline D5 public key** (`config::load_directory_pub` → `DirectoryVerifier::verify_binding`) before its `sig_pub` is trusted to verify a manifest/genesis. A binding that fails (forged/tampered/expired/rolled-back) yields a sanitized `untrusted` error and the read fails closed. The server is only the transport; it holds no D5 private key and cannot make the client accept a non-D5 author. Proven by the e2e forged-binding gate (a non-D5 signature → `untrusted`) and the unit tests in `directory.rs`. | Info | Accepted (correct). |
| 3 | **Requested-id integrity binding** | **Was Critical, now Sound (✓).** The download core binds a served record to the request via `manifest.file_id != ctx.file_id ⇒ FileIdMismatch`. Task 9's first cut built the viewer's `VerifyContext` with `file_id = manifest.file_id` (the **server-supplied** value), making that check a tautology — under the untrusted-server model a malicious server could serve, for requested id X, any other validly-signed record Y the user can decrypt, returned labeled as X (undetected content substitution). The code-quality review caught it; the fix validates the requested id (`hex16(&req.file_id)`) and passes **that** `Id` as `ctx.file_id`, matching the already-correct `decrypt_card`. The e2e gate 5 proves a bundle re-verified against a different `ctx.file_id` is rejected with `FileIdMismatch`. | **Critical (fixed)** | Resolved in commit `92f0496`; regression-gated by `browse_view_e2e.rs`. |
| 4 | **Unlocked-identity handling** | **Sound (✓).** `decrypt_card`, `open_content`, and `search_local` borrow the unlocked `Identity` **under the session lock** across a **synchronous** verify/derive step (no `take()`, no transient `None` window for a concurrent command to observe, no `await` held while borrowed). An initial Task-7 cut used take/restore (a `None` window); it was refactored to the held-guard borrow before acceptance. | Info | Accepted (correct). |
| 5 | **Local search index at rest (D-F)** | **Sound (✓).** Titles/tags are plaintext content, so the persisted index (`<dir>/index/search.idx`) is sealed with AES-256-GCM under a key derived by domain-separated HKDF-SHA256 from the unlocked identity's encryption secret (`hkdf_sha256_32(enc_secret, "MaxSecu-search-index-v1")`), with a fresh random 12-byte nonce per write (`nonce ‖ ct`). Tests prove: round-trips for the same identity; a different identity cannot open it; the plaintext title never appears on disk; an absent file → empty index while a **present-but-unreadable** (corrupt/foreign) file fails closed (tampering is not masked as "empty"). The derived key and the serialized plaintext are `Zeroizing`. Only `SearchHit`s of matches leave the TCB. | Info | Accepted (correct). |
| 6 | **Fail-closed decode / no oracle** | **Sound (✓).** All server-supplied bytes are parsed fail-closed: malformed §8.5 JSON → sanitized `fetch_failed` (no panic — attacker-controlled `chunk_count` no longer drives an eager `Vec::with_capacity`, fixed in Task 5); a verify-ladder failure → `verify_failed`; an unverifiable author → `untrusted`. No error distinguishes "absent" from "forbidden"/"unverifiable" in a way that leaks an oracle. | Info | Accepted (correct). |
| 7 | **UI rendering (XSS surface)** | **Sound (✓).** Dynamic content (title, tags, `author_fp`, blog body, error messages) is written via `textContent`/`setAttribute`/`createElement`, never `innerHTML` with interpolation (the only `innerHTML` is static shells). Blog content renders into a `<pre>` via `textContent` — the real XSS defense; a backend control-char strip is defense-in-depth. Images render via `data:image/png;base64,…` URLs, permitted by a narrowly-scoped CSP (`img-src 'self' data:`); nothing else in the CSP was loosened. | Info | Accepted (correct). |
| 8 | **Concurrency of authenticated commands** | **Sound (✓).** Each authenticated browse/view command re-auths on a fresh channel via `reauth`, which `try_lock`s a single `ConnectLock` and borrows the one non-`Clone` identity — so concurrent invocations cannot run. The UI therefore routes all per-card `decrypt_card` and the viewer's `open_content` through a shared FIFO `serial()` queue (one at a time, a failing task does not stall the queue, the caller still receives its own result), so the feed grid cannot spuriously fail with "busy". | Info | Accepted (correct). |

## 3. Threat-model coverage (Phase-3-relevant)

- **Content substitution by a hostile server (§3):** closed — the requested-id binding (after the Task-9 fix) rejects a substituted record; e2e gate 5.
- **Forged/rogue directory author:** closed — client re-verifies every binding under the pinned D5; a non-D5 author is `untrusted`; e2e gate 4.
- **Tampered/truncated/forged content:** closed — the unchanged §12.5 ladder (manifest/genesis/grant/DEK-commit + per-stream digest + AEAD) rejects splice/truncation/forgery; the header-only path inherits the same `verify_header`.
- **At-rest disclosure of browse metadata:** closed — the search index is encrypted under an identity-derived key; plaintext titles never hit disk; a foreign identity cannot read it.
- **Client DoS via malicious framing:** mitigated — no attacker-controlled preallocation; framing is bound-checked before allocation in the core.
- **XSS via attacker-influenced titles/tags/blog:** closed — textContent rendering; scoped CSP.

## 4. Residuals / deferrals (intentional, security-neutral)

- **First-contact directory resolution is TOFU** (`verify_binding`), not the opt-in key-transparency-gated `verify_binding_with_kt`. This matches the existing client-core design (the KT inclusion gate is opt-in, Phase 7); a split-view at genuine first contact is the residual TOFU accepts. Lighting up the KT gate for browse/download first-contact is a future opt-in, not a regression.
- **`client-core` rustfmt drift.** `client-core` carries pre-existing Phase 0–7 default-rustfmt drift (out of scope, as in Phases 1–2 for the server). Phase-3's new `client-core` lines were matched to the surrounding in-file style (the sibling `verify_and_stream_content`) rather than reformatted in isolation, which would have made identical lines inconsistent. `client-app`, `ceremony-harness`, and the UI are fmt-clean.
- **Search index O(N) reload per card** (O(N²) over a full browse) — accepted for Phase 3's sequential browse; an in-RAM cached index is a future optimization.
- **DRY follow-up:** factor the shared `verify_header` + small-stream loop common to `verify_and_open_headers` and `verify_and_stream_content` into one private helper, to avoid silent divergence in the TCB if a future hardening fix lands in only one (tracked; to be done as a deliberate, separately-reviewed refactor).

## 5. Conclusion

**PASS.** Phase 3 adds no server cryptography and keeps the secret-free, zero-knowledge, client-re-verified model intact. The browse/view path verifies every author under the pinned D5, binds every read to the **requested** file id (the one Critical, found in review and fixed, with a regression gate), runs the unchanged fail-closed §12.5 ladder, returns only render-ready data across the TCB seam, renders without an XSS surface, and keeps the local search index encrypted at rest. No Critical/High/Medium issues remain; the residuals are documented and security-neutral.
