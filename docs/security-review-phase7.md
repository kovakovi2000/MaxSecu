## 1. Methodology

The Phase-7 surface is the highest-value crypto/TCB code in the system (a zero-knowledge store gaining post-quantum confidentiality, threshold key custody, and a first-contact equivocation defense). Each focus area below was reviewed against an adversary who controls the server, the wire, and arbitrary attacker-supplied bytes, looking specifically for: re-binding / cross-protocol confusion, nonce reuse, missing zeroization, panics / OOB on attacker lengths (fail-open), allow-list bypasses (empty-pin "validates"), secret-dependent control flow, suite-downgrade, supply-chain regressions (new C / second TLS stack), and error oracles. Both build targets were taken green (§5).

### Additional Info-level observations (no action)

- **KT gossip-pin advance vs. binding fate.** `verify_binding_with_kt` advances the persisted KT checkpoint when the checkpoint's **own** signature + consistency + inclusion checks pass, *independently* of the later per-binding D5-signature / validity-window checks. This is intentional and safe: a checkpoint that verified under a pinned KT key and is a consistent extension is a legitimate gossip anchor regardless of any single binding's downstream fate, and the leaf was genuinely included — there is no poisoning vector. Documented in the function's doc comment.
- **Genesis-anchor encoding asymmetry.** `POST /v1/genesis-anchor` takes the `file_id` as base64; the `GET …/{file_id}` path takes it as hex (path-safe). Both are length/charset-validated to `400`. Cosmetic; no security impact.
- **`ml-kem` duplicate-transitive-deps.** `cargo deny check bans` emits `warning[duplicate]` for several transitive crates (`getrandom`, `rand_core`, `digest`, …) pulled in older minor versions by `ml-kem v0.3.2`. These are **warn-level**; the authoritative `cargo deny check` is green (exit 0). Recorded as an Info residual (§4).

## 3. Threat-model coverage (Phase-7-relevant rows)

- **No post-quantum confidentiality / harvest-now-decrypt-later (R6/D20, §15.2/§15.3):** addressed for **V2 uploads** by the hybrid X25519+ML-KEM-768 wrap — a CRQC must break **both** legs. V1 / un-migrated files remain the residual until the fleet PQ-enrolls (the fleet-wide re-enrollment ceremony is the §4 ops deferral, runbook `pq-reenrollment.md` ready).
- **Stolen offline recovery device / single cold copy (D6, §3.1/§16.3):** no longer total — the recovery private key is split K-of-N with Shamir; any `k-1` shares reveal nothing, so decrypt-everything now requires a threshold of custodians to convene. Residual: brief air-gapped-RAM reassembly at reconstruction.
- **First-contact directory equivocation / split view (#5/D14, §7.4):** closed by the directory KT log + client inclusion/consistency verification (`SplitView`/`Regression` fail-closed), the live first-contact defense that replaces the removed peer-pinning.
- **R27/D28 key-compromise cutoff over the real sink:** the genesis-anchor position basis is now real over the HTTP sink (one global counter ordering control appends vs. genesis anchors), idempotent and admin-gated.

## 4. Residuals / deferrals (intentional)

- **Hybrid combiner reassembly — N/A.** The hybrid wrap derives one KEK directly from both shared secrets; no shared secret or key is ever reconstructed from parts, so there is no reassembly residual for the KEM.
- **Shamir RAM-reassembly (§16.3).** The recovery scalar is briefly reassembled in air-gapped RAM at reconstruction (and exposed once at split). This is the accepted reconstruct-to-use custody posture, not a never-reassemble threshold cryptosystem; all transient copies are zeroized.
- **Client-side R27-comparison over the real sink — deferred.** The genesis-position **producer** is real over the HTTP sink (P7.8) and the R27/D28 cutoff is proven via `MemoryAuditSink` plus the real genesis anchor; wiring the client-side `CompromiseCheck` to *fetch* the genesis position over the real sink is the remaining ops integration (the proof shapes and the comparison logic are complete).
- **Fresh-per-process KT log key — ops-pin item.** The in-repo `sink-server` generates a fresh directory-KT log key per process; the real deployment pins a long-lived KT key (and gossips checkpoints to an independent witness/notary). The in-repo producer is the swap-in point; clients already pin `dir_log_public()`.
- **`ml-kem` `multiple-versions` deny WARN noise.** Duplicate transitive minor versions surfaced by `ml-kem` (`getrandom`/`rand_core`/`digest`/…). Warn-level only; `cargo deny check` is green. Accepted until the upstream minor-version skew settles.
- **Fleet-wide PQ re-enrollment ceremony** (runbook `docs/runbooks/pq-reenrollment.md` ready), a **real third-party KT witness/notary + vendor WORM/SIEM** (the in-repo producers are the swap-in), a **real CI runner + Authenticode certificate**, **ffmpeg video**, **real Dropbox**, and **P3.10 zstd** — all carried forward from Phase 6 as ops/future, not Phase-7 exit-gate items.
