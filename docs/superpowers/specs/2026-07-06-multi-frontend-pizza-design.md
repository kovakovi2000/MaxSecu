# Multi-Frontend Support + Cheese-Pizza Frontend ZIP — Design

**Date:** 2026-07-06
**Status:** Approved (design), pending implementation plan
**Area:** `crates/client-app/ui/` (WebView layer only — no backend/TCB change)

## 1. Goal

Let the MaxSecu client host **three genuinely distinct frontends**, each with its own
CSS and structural HTML, all driven by the **same Rust backend** and switchable from
the Appearance settings. One of the three is a full **cheese-pizza** visual redesign
that uses the supplied `pizza.png` on the login page.

Two deliverables:

1. **Scaffold the multi-frontend machinery** in the client (registry, runtime switcher,
   decoration slots, settings selector, build + tests). Build a working `default`
   frontend (today's design, verbatim), a wired-but-empty `slot3`, and a **stub**
   `pizza` frontend.
2. **Produce a design-brief ZIP** (like prior Codex rounds) that an external design
   agent uses to author the real cheese-pizza CSS + decoration, which then drops into
   the `pizza` slot.

The actual pizza CSS/decoration is **not** authored in this work — it is produced
externally from the ZIP and dropped in later.

## 2. Current architecture (starting point)

- Vanilla **TypeScript + Web Components** SPA. `index.html` mounts `<app-shell>`; all
  DOM is generated by TS components under `src/components/`. Single token-driven
  `styles.css` (~88KB). Build: esbuild bundles `src/main.ts` → `dist/main.js` and copies
  `index.html` + `styles.css`; the Tauri exe embeds `ui/dist/` at compile time.
- Today's "themes" (`tech`/`cheese`/`pottery`) are **color-token swaps only** via
  `data-theme` on `<html>` (`src/core/settings.ts` `getThemePreset/setThemePreset/
  applyThemePreset`; selector in `settings-screen.ts`). Same markup, same layout — the
  settings hint calls them "placeholders for upcoming visual passes."
- Backend contract and security/a11y invariants are documented in
  `ui/FRONTEND-GUIDE.md` and enforced by the Rust core + tests. This work must not
  touch `src/core/rpc.ts`, `types.ts`, command/event names, argument shapes, or the
  serial queue.

## 3. The "frontend" concept

A **frontend** is a named, self-contained visual skin with three parts:

- **id + label** — a registry entry.
- **A complete stylesheet it owns** — `styles.<id>.css` (a full standalone file, not a
  cascade override). Chosen over a shared-base + override approach so each frontend is
  self-contained and the external pizza designer owns exactly one file. Spacing/layout
  parity is enforced **by instruction** in the brief (preserve the spacing/layout tokens
  and component structure), not by a shared base.
- **An optional decoration module** — `frontends/<id>/deco.ts`, exporting
  `mount(root)` / `unmount()`, that injects frontend-specific decorative DOM into named
  slots (see §5).

Registered frontends:

| id        | label         | stylesheet         | deco module          | status                    |
|-----------|---------------|--------------------|----------------------|---------------------------|
| `default` | Default       | `styles.default.css` | none               | today's `styles.css`, verbatim |
| `pizza`   | Cheese Pizza  | `styles.pizza.css`   | `frontends/pizza/deco.ts` | **stub** `@import`s default (real one from ZIP) |
| `slot3`   | (3rd slot)    | `styles.slot3.css`   | none               | stub `@import`s default (falls back to Default look) |

Only **one** frontend stylesheet is active at a time. Components, backend, and
interactions are shared and unchanged across all three.

## 4. Runtime switching — `src/core/frontends.ts`

A new module replaces the theme-preset bits currently in `settings.ts`
(`getThemePreset` / `setThemePreset` / `applyThemePreset`). It exposes:

- `type FrontendId = "default" | "pizza" | "slot3"`
- `FRONTENDS: readonly { id: FrontendId; label: string }[]` — the registry.
- `normalizeFrontend(value: unknown): FrontendId` — defaults to `"default"`.
- `getFrontend(): FrontendId` — reads `localStorage["maxsecu.frontend"]`.
- `setFrontend(value): FrontendId` — persists + applies.
- `applyFrontend(value?): FrontendId` — the effecting function:
  1. rewrite `<link id="frontend-css">` `href` to `styles.<id>.css`,
  2. set `data-frontend="<id>"` on `document.documentElement`,
  3. call `unmount()` on the previous deco module (if any) then `mount()` on the new
     one (deco modules are looked up from a static map so esbuild bundles them),
  4. persistence handled by `setFrontend`.

Persistence stays **UI-local** in `localStorage` (`maxsecu.frontend`), exactly like the
current preset — no backend/`Settings` change. The existing `appearance.theme` (dark)
backend setting is untouched.

`applySettings()` in `settings.ts` keeps applying a11y data-attrs and pool sizing; its
`applyThemePreset()` call is replaced by `applyFrontend()`.

### Boot / no-flash

`index.html` ships a single `<link id="frontend-css" href="styles.default.css">` and a
tiny inline bootstrap script (runs before `main.js`) that reads
`localStorage["maxsecu.frontend"]`, and if valid + not `default`, rewrites the link
`href` and sets `data-frontend` before first paint. `data-theme="tech"` on `<html>` is
replaced by `data-frontend="default"`.

## 5. Decoration slots ("own HTML structure" hook)

Shared components expose a small number of **stable, empty mount points** so a frontend
can add structural/decorative DOM without forking components:

- `data-deco-slot="login"` — on the connect/register/recovery-login screens' root.
- `data-deco-slot="app-bg"` — a body-level layer in `app-shell` for full-page motifs.
- `data-deco-slot="header"` — in the app header.

Default and `slot3` leave these empty. The `pizza` deco module fills the `login` slot
with `pizza.png` and adds cheese-drip layers (styled by `styles.pizza.css`). Deco
modules must be **idempotent** and clean up on `unmount()` (used when switching
frontends live). Adding these empty slots is the only change to shared components and
must not alter layout/spacing when empty.

## 6. Settings UI — `settings-screen.ts`

The Appearance → **Theme** `<select>` (tech/cheese/pottery) becomes a **Frontend**
`<select>` populated from `FRONTENDS`: *Default / Cheese Pizza / (3rd slot)*. Same
wiring as today: read `getFrontend()` on render, call `setFrontend(value)` on change,
live-applies. The "placeholder" hint text is updated. The `settings-screen.test.ts`
expectations are updated to match.

## 7. Build + tests

- **Rename:** the current `styles.css` becomes `styles.default.css` (verbatim; it *is*
  the default frontend). Add stub `styles.pizza.css` and `styles.slot3.css`, each of
  which is a header comment plus `@import url("styles.default.css");` so the stub
  frontends render **exactly like Default** until authored (matches the intent that
  `slot3` falls back to Default styling, and keeps the `pizza` slot presentable before
  the external design lands). The external pizza design **replaces** `styles.pizza.css`
  wholesale with its own complete stylesheet (dropping the `@import`).
- **Build script** (`package.json`): copy `index.html`, all `styles.*.css`, and
  `pizza.png` into `dist/`; deco modules are bundled into `main.js` via the static map,
  so no separate copy needed. Verify `dist/` contains all three stylesheets + the asset.
- **`pizza.png`** is copied into `ui/` (e.g. `ui/assets/pizza.png`) as a tracked asset
  and into `dist/` at build.
- **Tests:** add `src/core/frontends.test.ts` (normalize defaults, persist round-trip,
  `applyFrontend` swaps link href + `data-frontend` attr, deco mount/unmount called).
  Update `settings-screen.test.ts`. Existing `npm test`, `npm run typecheck`, and
  `npm run test:a11y` must stay green (a11y structure is unchanged; empty deco slots are
  inert).

## 8. The ZIP (external design handoff) — "similar as before"

Produce `maxsecu-pizza-frontend.zip` (built to the repo root for download) containing:

- The `ui/` **source snapshot** excluding `node_modules/` and `dist/` — gives the
  external agent the components, `styles.default.css` to start from, the build, and the
  tests.
- `pizza.png` at a known path.
- **`FRONTEND-GUIDE.md`** (existing binding rules) **plus a new "Multi-frontend
  architecture" section** documenting the exact drop-in contract: the frontend registry,
  the `styles.pizza.css` + `frontends/pizza/deco.ts` (`mount`/`unmount`) interface, and
  the available `data-deco-slot` mount points.
- **`PIZZA-BRIEF.md`** (new task doc, paste-ready like `CODEX-REFINE-BRIEF.md`): build
  the cheese-pizza frontend — dripping-cheese / pizza motif; `pizza.png` on the login
  page; **spacing, layout, and component structure must stay identical** (only color,
  texture, and decoration change); deliver into `styles.pizza.css` +
  `frontends/pizza/deco.ts`; preserve backend contract, security, and a11y invariants.

## 9. Out of scope / non-goals

- Authoring the real cheese-pizza CSS/decoration (produced externally from the ZIP).
- Any backend, `types.ts`, RPC, or `Settings`-shape change.
- Changing component behavior, layout, or spacing (slots are inert when empty).
- The `slot3` visual design (intentionally empty until a future ZIP round).

## 10. Acceptance criteria

- Switching Frontend in settings live-swaps the active stylesheet + `data-frontend` and
  mounts/unmounts the deco module; choice persists across reloads with no first-paint
  flash.
- `default` renders identically to today. `pizza`/`slot3` stubs load without errors and
  render like Default (via `@import`) until authored.
- `npm run build` emits `dist/` with all three `styles.*.css` + `pizza.png`; `npm test`,
  `npm run typecheck`, `npm run test:a11y` all green.
- `maxsecu-pizza-frontend.zip` exists at repo root and contains the ui snapshot,
  `pizza.png`, updated `FRONTEND-GUIDE.md`, and `PIZZA-BRIEF.md`.
