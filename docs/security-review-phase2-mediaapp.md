# Phase 2 (Media App) — Security Review & Sign-off (Bootstrap, first-admin, directory publish, admin authz)

**Scope:** the Phase-2 change set on branch `media-app` — commit range `a77bd718..HEAD` (`a77bd718` = the Phase-1 baseline). The review targets the new **server** surface that the media app's bootstrap/admin flows added on top of the secret-free, zero-knowledge MaxSecu core, plus the test-only ceremony harness. The new client/UI code is in-scope only insofar as it consumes these endpoints; the cryptographic boundary it relies on (client-side re-verification of every served binding) is unchanged from Phase 0–7.

The additive server endpoints under review:

- `POST /v1/bootstrap` — first-run glass-break / first-admin registration.
- `POST /v1/directory` — directory-binding publish (D5-signature-gated).
- `GET /v1/pending` — the admin approval queue.
- `POST /v1/vouchers` — admin-issued enrollment vouchers.
- the `AdminSession` authz extractor + the migration of `post_control` (revocations / reinstatements / key-compromise) onto it.

**Verdict:** **PASS** — no Critical, High, or Medium findings. The additive endpoints preserve the secret-free / zero-knowledge / client-re-verified model: the only new server-side cryptography is **signature verification against a pinned public key**. One Info-level deferral is documented (§4). No fix commits were required by this review.

---

## 1. Methodology

The new endpoints were reviewed against an adversary who controls the server operator console *partially* (knows no offline D5 private key), the network, and arbitrary attacker-supplied request bytes, looking specifically for: privilege escalation (self-minted admin), directory pollution (server forging or accepting an unsigned binding), bootstrap-window abuse / race, secret oracles (length/timing on the bootstrap secret), and error oracles on the new routes. The authoritative reference is `DESIGN.md` §4.2 (coarse server authz), §4.5 (offline ceremony / operator console), §10 (admin model), and api.md §3/§6. Verification artifacts: `crates/server/tests/sanitized_errors.rs`, `crates/server/src/store.rs::phase2_store_tests`, `crates/server/src/http.rs` in-crate tests, `crates/server/tests/directory_e2e.rs`, and the Phase-2 exit-gate e2e `crates/client-app/tests/bootstrap_admin_e2e.rs` (full bootstrap → first-admin → voucher-enroll → pending → ceremony-approve → valid recipient, over real loopback TLS 1.3).

## 2. Per-area findings & dispositions

| # | Area | Finding | Severity | Disposition |
|---|---|---|---|---|
| 1 | **`POST /v1/bootstrap`** | **Sound (✓).** Double-gated: (a) it is valid **only while the bootstrap window is open** — i.e. `Store::has_any_binding()` is `false`, no directory binding has ever been published; the window slams shut on the first publish (`409 Conflict` thereafter, proven by the e2e's "bootstrap closes after the first publish" assertion); and (b) the request must carry the operator-console bootstrap secret, which the server holds **only** as `SHA-256(secret)` (`AuthConfig::with_bootstrap_secret_hash`) and compares **hash-to-hash** — equal-length 32-byte digests, so there is no practical length or timing oracle on the real secret. Crucially, bootstrap **creates a user and never confers admin** — exactly like the ordinary `POST /v1/users` path. Admin power exists only once the offline ceremony D5-signs that user's binding with `Role::Admin` and it is published (D-K). | Info | Accepted (correct). The first-run race (whoever races the open window) is mitigated by the operator-only console secret (§4.5): an attacker without it cannot register through the window, and even a winner gets only an ordinary unsigned account until the offline ceremony signs it. |
| 2 | **`POST /v1/directory` (publish)** | **Sound (✓).** The handler stores a binding **only if it verifies under the pinned D5 public key** (`AuthConfig::with_directory_pub`) — an anti-pollution gate. The server holds **only the public half** of D5: it can *verify* a binding, never *forge* one. A binding that fails D5 verification is rejected (no row written). The route is **unauthenticated by design**: the authority is the D5 signature, not a session — bootstrap admins' own bindings must be publishable *before any admin session can exist* (chicken-and-egg), and the signature is strictly stronger than a session token. This endpoint is **not** the security boundary: the client `DirectoryVerifier` independently re-verifies every served binding against the **pinned root** before treating a key as a recipient (proven end-to-end by `bootstrap_admin_e2e.rs::full_bootstrap_to_valid_recipient` and `directory_e2e.rs`). The server gate is defense-in-depth that keeps junk out of the table; even a buggy/hostile server cannot make the client accept a non-D5-signed binding. | Info | Accepted (correct). The e2e cosmetic fix in this task switched the approval publish call's bearer to `None`, matching the unauthenticated-by-design contract. |
| 3 | **Admin authz (`AdminSession`)** | **Sound (✓).** An admin is defined as **a D5-signed directory binding carrying `Role::Admin`, within its validity window**, verified server-side against the pinned D5 key. `AdminSession` resolves the caller's session → `user_id` → their latest published binding → checks the D5 signature + validity window + `Role::Admin`. This unifies all admin gating: the pending queue (`GET /v1/pending`), voucher issuance (`POST /v1/vouchers`), **and** the control-log writes (`post_control`: revocations / reinstatements / key-compromise), which were migrated onto it. The server can **recognize** admin (verify a pinned-public-key signature) but can never **confer** it (it has no D5 private key) — D-K. This replaces the advisory `users.roles` table gate with a cryptographically-grounded one; strictly stronger. | Info | Accepted (correct). |
| 4 | **Zero-knowledge / secret-free invariant** | **Sound (✓).** No Phase-2 endpoint introduces a server-held secret, private key, salt, KDF parameter, or plaintext. The server stores opaque, client-verified bytes (the binding is `canonical(DirBinding)` + a 64-byte D5 signature; vouchers are stored as `SHA-256` hashes; the bootstrap secret as `SHA-256`). The **only** new server cryptography is signature **verification against a pinned public key** (coarse authz, §4.2/§10) and hash comparison — no signing, no decryption, no secret material. | Info | Accepted (correct). |
| 5 | **Sanitized errors / no oracle** | **Sound (✓).** The new endpoints use the existing uniform error shapes (`401`/`403`/`404`/`409`/`500`, bare status + empty body; no detail, no existence oracle). `crates/server/tests/sanitized_errors.rs` was extended with the new `Store` methods (`has_any_binding`, `list_pending_users`, `issue_voucher`) on the fault-injecting stores and asserts the new paths leak none of the forbidden internals and preserve the unknown-vs-unauthorized indistinguishability. | Info | Accepted (correct). |
| 6 | **Test-only ceremony harness** (`tools/ceremony-harness`) | **Sound (✓).** `maxsecu-ceremony-harness` scripts the offline D5 ceremony **in-process** purely so tests can drive a realistic bootstrap → approve flow. It is explicitly security-degraded / test-only and is **never** a production path: the real ceremony runs the air-gapped CLIs with the D5 private key kept offline (DESIGN §4.5). It is wired only into `[dev-dependencies]`/test targets; no production binary links it. | Info | Accepted (correct). |

## 3. Threat-model coverage (Phase-2-relevant rows)

- **Self-minted admin / privilege escalation (D-K, §10):** closed. No endpoint grants admin; admin authority flows *only* from a D5-signed `{User, Admin}` binding the server can verify but not forge. Bootstrap creates ordinary users; the window is secret-gated and one-shot.
- **Directory pollution / forged binding (§6, anti-pollution):** the publish gate rejects anything not D5-signed; and even if bypassed, the client re-verifies against the pinned root, so a forged binding is never treated as a recipient.
- **Bootstrap-window abuse / first-run race (§4.5):** the window is open only until the first publish and is additionally guarded by the operator-console bootstrap secret (stored hashed); a racer without the secret cannot enter, and a racer with it still gains only an unsigned account.
- **Error/existence oracles (§16.2):** the new routes reuse the constant-shape sanitized responses, covered by the extended `sanitized_errors.rs` suite.

## 4. Residuals / deferrals (intentional)

- **Coarse admin gate does not yet honor a role-narrowing de-admin tombstone.** A de-admined admin retains server-side **coarse** powers until their binding is re-signed (without `Role::Admin`) or expires. This is **acceptable, not a hole**, per DESIGN §4.2/§10: the server's coarse authz is *not* the security boundary. The de-admin tombstone's authoritative effect is **client-side + sink-anchored** — clients reject a de-admined admin's signed actions regardless of what the server's coarse gate allows. Server-side tombstone honoring (narrowing the `AdminSession` gate on an observed de-admin record) is a follow-up; it tightens defense-in-depth but changes no security guarantee.

## 5. Conclusion

**PASS.** The Phase-2 additive endpoints (`POST /v1/bootstrap`, `POST /v1/directory`, `GET /v1/pending`, `POST /v1/vouchers`) and the `AdminSession` authz model (including the `post_control` migration) keep the secret-free, zero-knowledge, client-re-verified model intact. The server gains only the ability to **recognize** authority (verify against pinned public keys) — never to **confer** it. No Critical/High/Medium issues are introduced; the single Info-level residual (server-side de-admin tombstone honoring) is a documented, security-neutral follow-up.
